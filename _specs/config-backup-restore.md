# Feature Spec: Config Backup and Restore

## Overview

Add a way for an operator to take a portable snapshot of all dashboard configuration — rooms, sensors and their per-device settings, blocked sensors, light groups, theme assignments, the Hue bridge pairing, and user credentials — and restore that snapshot later or on a different Warren host. Time-series sensor history (InfluxDB) is intentionally outside the scope; the backup is for "what is the dashboard configured to do," not "what has it observed."

## Goals

- Let an operator capture the full editable configuration of a Warren installation as a single downloadable file.
- Let an operator restore from that file on the same host (after a misconfiguration, dropped DB, host rebuild) or on a new host (migrating to new hardware) without losing room layout, sensor assignments, light groups, or the Hue pairing.
- Make the snapshot stable across minor schema changes by versioning the file and rejecting incompatible versions with a clear error.
- Make snapshots safe to share between trusted operators of the same household — no leakage of broker passwords or InfluxDB tokens, since those are infrastructure secrets owned by `warren setup`.

## Non-Goals

- **No time-series export.** InfluxDB `sensor_readings` are large, replayable from devices, and out of scope. Sensor history is not backed up or restored.
- **No camera frame archive.** Cameras are stateless from the dashboard's point of view; only their assignment to rooms is config and is therefore included.
- **No firmware backup.** ESP32 device firmware and NVS-stored config live on the device and are reproducible from `firmware/` source.
- **No backup of infra-level secrets.** MQTT broker password, InfluxDB admin token, `iron-session` sealing password, and the Caddy TLS material are owned by `warren setup` and stay out of the snapshot. A restore on a fresh host expects `warren setup` to have already run.
- **No automated scheduled backups in this iteration.** Manual export only; cron-style scheduling is a follow-up.
- **No partial / per-room restore in this iteration.** Restore is whole-snapshot.

## User Stories

- As an operator, I want to export my current dashboard configuration to a single file, so that I can keep a copy somewhere safe before making big changes.
- As an operator, I want to import a previously exported file on a fresh Warren install, so that I get back all my rooms, sensors, light groups, and Hue pairing without re-clicking the setup flow.
- As an operator, I want a clear preview of what a snapshot contains before I restore it, so that I do not destroy current config by accident.
- As an operator, I want the restore to refuse to run on an incompatible snapshot version, so that I get an honest error instead of a silently broken dashboard.
- As an operator, I want to keep my current login working after a restore, so that I am not locked out of my own dashboard if the snapshot was taken under different credentials.

## Functional Requirements

### Snapshot creation

- A signed-in user can trigger an export from the dashboard. The result is a single file the browser downloads.
- The export captures every row from the configuration tables in `nextjs-ui/lib/server/db.ts`: `rooms`, `sensors`, `sensor_config`, `sensor_announcements`, `blocked_sensors`, `hue_bridge`, `hue_devices`, `hue_light_state`, `light_groups`, `light_group_members`, `users`.
- The export does **not** include InfluxDB content, the `iron-session` cookie key, MQTT/InfluxDB credentials from `nextjs-ui/.env`, or the Caddy TLS material under `docker/tls/`.
- The export includes a small header with: snapshot schema version, Warren app version, the host's stable identifier (or null), the export timestamp, and a count of rows per table.
- The export is a JSON document. Binary fields are base64-encoded. The file is suggested with a name like `warren-snapshot-<yyyy-mm-dd-hhmm>.json`.
- Hue device records and the bridge `app_key` are included so that the restored host can keep talking to the same bridge without re-pressing the pairing button. The `hue_light_state` rows can be regenerated by the poller and are included only as a "last-known" cache for cosmetic continuity.
- The export operation is read-only. It must not mutate any table.

### Snapshot storage

- The dashboard does not store snapshots server-side in this iteration. Exports stream straight to the browser as a download.
- The snapshot file format is documented in `nextjs-ui/lib/server/backup/README.md` (created with the feature) so external tooling can read it.

### Restore

- A signed-in user can upload a snapshot file from the dashboard. Before any DB write, the server parses the file and returns a preview that lists row counts per table, the snapshot's schema version, the export timestamp, and any incompatibilities.
- The user must confirm an explicit "Replace current configuration" action to proceed. Confirmation requires re-typing a short phrase or clicking through a non-default-styled "destructive" confirm dialog.
- Restore is **whole-snapshot replace**: every targeted table is wiped and repopulated transactionally. If the restore fails mid-way, the transaction rolls back and the dashboard is left in its pre-restore state.
- The `users` table merge rule: the **currently signed-in user is preserved**. If the snapshot contains a user with the same username as the active session, that user's password hash from the snapshot is **not** applied; instead the active session's user keeps its existing password. All other users in the snapshot are imported as-is. This prevents an operator from accidentally locking themselves out by restoring a snapshot taken with a different password.
- After restore, the Hue runtime is asked to re-sync against the imported `hue_bridge.app_key`. If the bridge is unreachable or the app key has been revoked, the restore still completes, the dashboard surfaces a "Hue bridge needs re-pairing" warning, and the rest of the configuration remains intact.
- After restore, the MQTT subscriber and Hue runtime are restarted in-process so that `instrumentation.ts`-managed subsystems pick up the new state without a server restart.

### Schema-version compatibility

- The snapshot file carries a numeric `schema_version`. A restore on a host whose code expects a different `schema_version` is refused with a message like "Snapshot was created with Warren schema vN; this host expects vM. Upgrade or downgrade Warren to match before restoring."
- A schema migration that changes table shape must increment `schema_version` and ship a migration step that knows how to upgrade older snapshots, or refuse them.

### Auth & access control

- Both export and restore require a valid `iron-session` cookie. They are NOT in any of the `proxy.ts` allowlists.
- Both endpoints are reachable only via the dashboard UI; there is no device-side equivalent.

## UI / UX

A new **Backup & Restore** card lives on a new **Settings** page (or, if a settings page does not yet exist, a new top-level sidebar entry visible only to authenticated users).

The card has two sections:

1. **Export.** A single primary button labelled "Download snapshot." Clicking it triggers the browser download. A small caption underneath summarises what is and is not included ("Includes rooms, sensors, lights, Hue pairing, users. Does not include sensor history, broker credentials, or TLS material.").

2. **Restore.** A file-picker for `.json` snapshots and a disabled-by-default "Restore" button. Once a file is chosen, the dashboard fetches a server-side preview and shows a summary table: schema version, export timestamp, row counts per table, and any compatibility warnings. The "Restore" button enables only when the preview is compatible. Clicking it opens a destructive-styled confirmation dialog (red Catalyst `Alert` with "This will replace your current configuration" copy and a typed-confirmation field). On success the page reloads to pick up the new state; on failure an error toast is shown with the reason.

The Catalyst conventions in `nextjs-ui/CLAUDE.md` apply: use `Button color="red"` for the destructive action, the existing `confirm-dialog.tsx` `Alert` pattern for the confirmation, and standard `Field` / `Input` / `Description` primitives for the preview.

## Data Model

No new persisted tables. Reads only from existing tables.

The on-disk JSON snapshot has roughly this shape (illustrative — exact field names finalised during implementation):

- `header` — `schema_version`, `app_version`, `host_id`, `exported_at`, `row_counts`
- `tables` — one entry per table listed under "Snapshot creation," each a list of row objects with column names matching SQLite columns

A single `schema_version` constant lives in shared code so that both the exporter and the restore validator agree on the current value.

## API

All under `/api/admin/backup/` (new prefix, new route handlers in `nextjs-ui/app/api/admin/backup/`):

- `GET /api/admin/backup/export` — returns the snapshot as a JSON download (`Content-Disposition: attachment`). Auth required.
- `POST /api/admin/backup/preview` — accepts a multipart upload of a snapshot file, parses it without mutating state, returns the header plus row-count summary plus a list of compatibility warnings. Auth required.
- `POST /api/admin/backup/restore` — accepts a multipart upload, performs the transactional restore, restarts in-process subsystems, returns success or a structured error. Auth required.

All three endpoints require an `iron-session` cookie and must be added to no allowlist in `proxy.ts`.

## Acceptance Criteria

- [ ] An authenticated user can download a snapshot file from the new Settings → Backup & Restore card.
- [ ] The snapshot includes every configuration table listed in "Snapshot creation" and excludes everything listed in "Non-Goals."
- [ ] An authenticated user can upload a snapshot, see a preview of what it contains, and trigger a restore from the same card.
- [ ] A restore replaces every targeted table inside a single SQLite transaction; a forced failure mid-restore leaves the prior configuration intact.
- [ ] After a successful restore, the currently signed-in user can still log in with their pre-restore password.
- [ ] After a successful restore, the dashboard reflects the restored rooms, sensor assignments, blocked sensors, light groups, themes, and Hue pairing without requiring a server restart.
- [ ] Restoring a snapshot whose `schema_version` does not match the running host fails with an explicit, user-facing error and no DB writes.
- [ ] Export and restore endpoints return 401 / redirect when called without a session.
- [ ] An export taken on Host A and restored on Host B (fresh `warren setup`, same Hue bridge on the LAN) results in a working dashboard, with the Hue runtime re-syncing successfully against the imported `app_key`.
- [ ] An end-to-end Playwright spec under `nextjs-ui/tests/e2e/` covers: export → wipe DB → restore → assert config matches.

## Open Questions

- Should restore offer a "merge" mode in addition to "replace," for the case where an operator wants to import another household's room template without losing their own? Out of scope for v1, but worth noting before locking the API shape.
- Should the snapshot be encrypted at rest with a user-supplied passphrase? It contains the Hue `app_key` and password hashes — sensitive but not catastrophic. A passphrase option could ship in a follow-up.
- Should we retain a small server-side ring buffer of automatic snapshots (for example, before each restore, or on app upgrade)? Useful for "undo" but adds disk-management surface; deferred.
- Should the Backup & Restore UI live on a new top-level **Settings** page, or under a new **Admin** sub-section? Depends on whether other operator-only settings ship in the same window.
- How do we deal with a snapshot that references a Hue bridge that has been factory-reset (so the `app_key` is invalid) on the same LAN? Probably: surface the warning, leave the bridge config in place, and let the user re-pair from the existing **Integrations → Hue** flow. Confirm before implementation.
